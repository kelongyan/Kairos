# 基于 RAG + Multi-Agent 的智能知识库系统

## 项目方案书

---

## 一、项目背景

随着企业和团队规模的扩大，内部知识资产（技术文档、产品手册、制度规范、会议纪要、培训资料等）呈指数级增长。这些知识通常散落在不同格式的文件中（PDF、Word、网页、PPT 等），存储在不同的平台和系统里，导致以下问题日益突出：

- **知识检索效率低**：员工需要在大量文档中手动搜索所需信息，耗时长、遗漏多
- **跨文档推理困难**：许多问题的答案分散在多份文档中，需要人工综合才能得出完整结论
- **知识利用率不足**：大量沉淀的经验与文档无人问津，新员工无法快速继承团队知识
- **信息准确性难以保证**：传统搜索只能返回文档链接，无法直接给出经过验证的精准回答

与此同时，大语言模型（LLM）虽然在自然语言理解和生成方面表现出色，但单独使用时存在知识过时和事实幻觉两大核心缺陷，无法直接用于对准确性要求较高的企业知识问答场景。

因此，本项目提出构建一套基于检索增强生成（RAG）与多智能体（Multi-Agent）协作的智能知识库系统，将企业私有文档转化为可被精准检索和智能问答的知识资产，同时通过多 Agent 协作机制处理复杂查询任务，实现"让知识找到人，而不是让人去找知识"的目标。

---

## 二、项目目标

### 2.1 总体目标

构建一个企业级智能知识库系统，实现文档的全生命周期管理（上传 → 解析 → 切片 → 索引 → 检索 → 问答），并通过多 Agent 协作架构提升复杂问题的回答质量与可靠性。

### 2.2 具体目标

| 目标维度 | 描述 |
|:---|:---|
| 知识管理 | 支持多格式文档的统一上传、解析、结构化管理和版本控制 |
| 精准检索 | 采用混合检索策略（语义 + 关键词），实现对知识库内容的高效、准确检索 |
| 智能问答 | 基于 RAG 架构，结合 LLM 生成能力与知识库检索结果，提供有依据、可溯源的精准回答 |
| 多 Agent 协作 | 通过多角色 Agent 分工（规划、检索、撰写、审核），提升复杂问题的处理质量 |
| 可观测性 | 记录完整的问答链路与 Agent 执行过程，支持调试、回溯与持续优化 |
| 可扩展性 | 架构设计支持知识库规模横向扩展、Agent 角色灵活增减、工具能力按需接入 |

## 四、功能需求

### 4.1 文档管理模块

#### 4.1.1 文档上传与解析

- 支持上传 PDF、Word（.docx）、Markdown、HTML、纯文本（.txt）等主流文档格式
- 自动识别文档格式并调用对应的解析引擎提取纯文本内容
- 支持批量上传与拖拽上传
- 上传时自动提取文档元信息（文件名、页数、字数、上传时间等）

#### 4.1.2 智能文本切片

- 支持多种切片策略：
  - **固定长度切片**：按指定 Token 数量切割，支持配置重叠（overlap）长度
  - **语义结构切片**：基于文档标题层级（H1/H2/H3）进行自然段落分割，保留段落间的逻辑结构
  - **递归切片**：先按大章节切分，再按子标题细分，维持完整的层级关系
- 每个切片携带结构化元数据，包括但不限于：
  - 所属文档 ID 与文件名
  - 在文档中的位置信息（页码、章节标题路径）
  - 文档类型标签（如"技术文档""产品手册""制度规范"）
  - 创建/上传时间
- 切片策略可按知识库级别独立配置，不同知识库可选择不同的切片参数

#### 4.1.3 向量化与索引构建

- 使用 Embedding 模型将文本切片转化为高维向量表示
- 向量与原始文本、元数据一并存入向量数据库
- 同时构建关键词倒排索引（BM25），支持精确关键词匹配
- 支持文档更新时的增量索引（新增切片入库、旧切片标记失效）

#### 4.1.4 知识库生命周期管理

- 支持创建多个独立知识库，每个知识库可设置名称、描述、访问权限
- 知识库内文档支持查看、替换、删除操作
- 文档删除时自动清理对应的向量索引与关键词索引
- 支持知识库级别的统计信息查看（文档总数、切片总数、最近更新时间等）

---

### 4.2 混合检索模块

#### 4.2.1 双通道检索

- **语义检索通道**：将用户查询向量化后在向量数据库中执行近似最近邻搜索（ANN），返回语义最相关的 Top-K1 切片
- **关键词检索通道**：基于 BM25 算法对倒排索引执行精确匹配检索，返回关键词最相关的 Top-K2 切片
- 两个通道并行执行，互不阻塞

#### 4.2.2 结果融合与重排序

- 对两个通道返回的候选结果进行去重合并
- 使用 Re-rank 模型（交叉编码器）对合并后的候选集进行二次精排，综合语义相关性和关键词匹配度
- 输出最终 Top-K 结果作为 LLM 的参考上下文

#### 4.2.3 检索增强策略

- **查询改写**：对用户原始查询进行意图理解和改写，消除歧义、补全上下文
- **多轮检索**：当首次检索结果不充分时，Agent 可自主发起追加查询（调整关键词或语义表述）
- **元数据过滤**：支持按文档类型、时间范围、所属知识库等条件对检索范围进行预过滤

---

### 4.3 Multi-Agent 协作引擎

#### 4.3.1 Agent 角色定义

系统内置以下专业化 Agent，各自承担独立职责：

| Agent 名称 | 角色定位 | 核心职责 |
|:---|:---|:---|
| Planner Agent | 规划者 | 接收用户查询，分析意图复杂度，决定是否需要拆解子任务，生成分步执行计划并调度其他 Agent |
| Retrieval Agent | 检索员 | 根据 Planner 分配的检索任务，调用混合检索模块从指定知识库中获取相关文档片段 |
| Search Agent | 调研员 | 当知识库内信息不足以回答问题时，通过外部搜索接口获取补充信息 |
| Analyst Agent | 分析师 | 对检索到的信息进行数据对比、趋势分析、逻辑推理等深度加工 |
| Writer Agent | 撰写者 | 整合所有 Agent 提供的证据材料，生成结构完整、逻辑连贯的最终回答 |
| Reviewer Agent | 审核者 | 对 Writer 生成的回答进行事实核查，验证每句话是否有来源依据，检测潜在的幻觉内容 |

#### 4.3.2 协作编排机制

- 采用**状态图（State Graph）**进行 Agent 工作流编排，支持：
  - **顺序执行**：Agent A 完成后触发 Agent B
  - **并行执行**：多个 Agent 同时处理不同子任务
  - **条件分支**：根据中间结果动态决定下一步走向
  - **循环回退**：Reviewer 审核不通过时，将任务回退给 Writer 重新生成
- Planner Agent 作为入口调度器，根据查询类型动态选择不同的 Agent 协作路径：
  - **简单查询**：Planner → Retrieval → Writer（跳过不必要的 Agent）
  - **复杂查询**：Planner → Retrieval + Search（并行）→ Analyst → Writer → Reviewer
  - **模糊查询**：Planner → Retrieval →（结果不足）→ Search 补充 → Writer → Reviewer

#### 4.3.3 Agent 推理过程记录

- 每个 Agent 在执行过程中记录完整的推理链路：
  - 接收到的输入（任务描述、上下文）
  - 做出的决策（选择了什么工具、为什么）
  - 产生的中间输出
  - 执行耗时与 Token 消耗
- 推理链路数据持久化存储，支持事后回溯与可视化展示

---

### 4.4 智能问答模块

#### 4.4.1 对话交互

- 支持多轮对话，系统自动维护会话上下文
- 用户可在对话中引用前文内容（如"刚才那个问题的第二点能展开说说吗"）
- 支持为每段对话绑定特定知识库范围（仅查询指定知识库）

#### 4.4.2 流式输出

- 回答采用 SSE（Server-Sent Events）流式传输，实现逐字/逐句实时输出
- 用户无需等待完整生成即可开始阅读回答内容

#### 4.4.3 问答溯源

- 回答中的每句话标注信息来源（对应的文档切片 ID）
- 用户可点击溯源标记，查看原文上下文（高亮显示被引用的具体段落）
- 溯源信息包括：来源文档名称、页码/章节、相关度评分

#### 4.4.4 对话记忆

- **短期记忆**：同一会话内维护完整的对话历史，支持上下文关联
- **长期记忆**：跨会话记录用户的关注领域、常用知识库、偏好设置，提升后续问答的个性化程度

---

### 4.5 可观测性与评估模块

#### 4.5.1 Trace 日志系统

- 为每次问答请求生成唯一的 Trace ID
- 完整记录请求生命周期内的所有关键事件：
  - 用户原始查询
  - 查询改写过程
  - 各 Agent 的调用顺序、输入输出、耗时
  - 检索命中的文档切片及其评分
  - LLM 的 Prompt 与响应内容
  - Token 消耗量与预估成本
- Trace 日志支持按时间、用户、知识库、Agent 等维度进行检索和过滤

#### 4.5.2 Agent 推理可视化

- 前端提供 Agent 执行流程的可视化回放界面
- 以流程图形式展示各 Agent 的调用关系（顺序/并行/回退）
- 每个节点可展开查看详细输入、输出和决策过程
- 支持按时间轴查看执行耗时分布

#### 4.5.3 评估指标体系

建立标准化的评测机制，用于持续衡量系统表现：

| 指标名称 | 定义 | 度量方式 |
|:---|:---|:---|
| 检索命中率（Hit Rate） | Top-K 检索结果中包含正确答案的比例 | 基于标准 QA 测试集自动评测 |
| 回答准确率 | 最终回答与标准答案的语义一致程度 | LLM-as-Judge 自动评分 + 人工抽检 |
| 幻觉率 | 回答中包含知识库无法支撑的信息的比例 | 逐句检查是否有有效溯源引用 |
| 溯源准确率 | 引用的来源切片是否真正支持对应表述 | 自动验证 + 人工抽检 |
| 端到端响应时延 | 从用户提交查询到回答完全输出的总时间 | P50 / P95 / P99 分位统计 |
| Token 消耗效率 | 单次问答的平均 Token 使用量 | 按 Agent 维度分类统计 |

- 支持自定义评测数据集的导入与管理
- 支持定期自动运行评测任务并生成趋势报告

---

### 4.6 用户与权限管理模块

#### 4.6.1 用户管理

- 用户注册与登录（账号密码 + 邮箱验证）
- 基于 JWT 的身份认证与会话管理
- 用户个人信息管理（头像、昵称、密码修改）

#### 4.6.2 权限控制

- 基于角色的访问控制（RBAC）：管理员 / 知识库管理员 / 普通用户
- 知识库级别的访问权限管理：
  - 公开知识库（所有用户可查询）
  - 受限知识库（仅授权用户/团队可查询）
- 敏感操作（如删除知识库、修改权限）需二次确认

#### 4.6.3 操作审计

- 记录用户的关键操作日志（文档上传/删除、权限变更、知识库创建等）
- 管理员可查看审计日志，支持按用户、操作类型、时间范围过滤

---

### 4.7 数据统计看板

- **平台级统计**：总用户数、总知识库数、总文档数、总切片数、累计问答次数
- **知识库级统计**：各知识库的文档增长趋势、问答热度排行、高频查询词云
- **问答质量统计**：平均响应时延趋势、检索命中率趋势、用户满意度评分分布
- **资源消耗统计**：Token 消耗量按日/周/月统计、各 Agent 的 Token 占比
- 支持图表化展示（折线图、柱状图、饼图、雷达图）

## 五、非功能性需求

### 5.1 性能要求

| 指标 | 目标值 |
|:---|:---|
| 单次混合检索耗时 | ≤ 200ms（P95） |
| 简单问答端到端响应时延 | ≤ 3s（P95） |
| 复杂多 Agent 问答响应时延 | ≤ 8s（P95） |
| 文档解析与切片吞吐量 | ≥ 50 页/分钟 |
| 系统并发用户支持 | ≥ 100 同时在线 |

### 5.2 可用性要求

- 核心服务可用性 ≥ 99.5%
- 支持服务异常时的优雅降级（如 LLM 服务不可用时，退化为纯检索模式返回相关文档片段）
- Agent 执行过程中某个环节失败时，不影响已完成的中间结果，支持从失败点恢复

### 5.3 安全要求

- 所有 API 接口需进行身份认证与权限校验
- 用户上传的文档在存储和传输过程中加密
- LLM 调用过程中对用户数据进行脱敏处理，防止敏感信息泄露
- 对外部工具调用（如 Search Agent 的互联网搜索）实施沙箱隔离和参数校验
- 高风险操作（如批量删除、权限变更）需审批确认

### 5.4 可维护性要求

- 各模块松耦合设计，支持独立部署和升级
- 提供完整的系统配置中心（LLM 模型选择、检索参数、Agent 策略等均可通过界面配置，无需修改代码）
- 所有配置变更有版本记录，支持回滚

## 六、系统架构概要

系统采用分层架构设计，从上到下分为四层：

| 层级 | 名称 | 职责 |
|:---|:---|:---|
| 第一层 | 表示层 | 提供 Web 前端界面，包括文档管理、对话交互、Agent 可视化、数据看板等模块 |
| 第二层 | 业务服务层 | 承载核心业务逻辑，包括文档管理服务、问答服务、Agent 编排引擎、用户管理服务等，各服务独立部署，通过 API 网关统一对外暴露 |
| 第三层 | AI 能力层 | 封装 LLM 调用、Embedding 向量化、混合检索、Re-rank 排序等 AI 基础能力，对上层提供统一接口 |
| 第四层 | 数据存储层 | 包括向量数据库（存储文档向量与语义索引）、关系型数据库（存储业务数据与用户信息）、缓存数据库（存储会话状态与高频查询缓存）、对象存储（存储原始文档文件） |

各层之间通过定义清晰的接口进行通信，支持独立扩展与替换。

## 七、后续可扩展方向

### 7.1 GraphRAG 知识图谱增强

在传统向量检索与 BM25 检索的基础上，引入知识图谱层：

- 对文档进行实体与关系抽取，自动构建领域知识图谱
- 利用图结构实现多跳推理（如"A 部门的负责人曾参与过哪些 B 类项目"）
- 通过社区检测算法（Community Detection）生成全局摘要，支持回答需要跨文档综合的宏观问题
- 适用于知识库规模超过数万份文档的大型组织

### 7.2 多模态文档支持

- 扩展文档解析能力，支持对文档中的表格、图片、流程图进行结构化提取
- 接入多模态 Embedding 模型，实现对图表内容的语义检索
- 支持用户以图片形式提问（如拍摄一张设备故障照片，系统从知识库中检索解决方案）

### 7.3 MCP 标准化工具接入

- 基于 MCP（Model Context Protocol）协议标准化工具接入层
- 支持通过配置而非代码开发，快速接入新的外部工具：
  - 企业内部系统（ERP、CRM、OA）
  - 数据库直连查询（SQL/NoSQL）
  - 代码仓库检索（Git）
  - 第三方 API 服务
- 构建工具市场，管理员可自主启用/禁用/配置各类工具

### 7.4 A2A 跨系统 Agent 协作

- 基于 A2A（Agent-to-Agent）协议实现与其他智能系统的 Agent 互联互通
- 支持跨系统的任务委派与结果回传（如本系统的 Planner Agent 将子任务委派给财务系统的专用 Agent）
- 构建企业级多系统协作的智能工作流网络

### 7.5 模型微调与自适应优化

- 基于系统积累的 Trace 日志与评测数据，对 LLM 进行针对性微调：
  - 优化特定领域的回答质量
  - 改进工具选择与任务拆解的准确性
  - 降低特定场景下的幻觉率
- 建立持续学习闭环：评测 → 发现问题 → 微调优化 → 重新评测

### 7.6 移动端与多渠道接入

- 开发移动端应用（iOS/Android），支持随时随地访问知识库
- 接入企业即时通讯工具（钉钉、企业微信、飞书）作为问答入口
- 支持语音输入提问与语音播报回答

### 7.7 智能推送与主动服务

- 基于用户角色和浏览历史，主动推送相关知识的更新通知
- 定期生成知识库洞察报告（如"本月被查询最多的主题""近期新增的高频问题"）
- 在新员工入职时，根据其岗位自动生成个性化的知识学习路径

### 7.8 协作与反馈机制

- 支持用户对回答进行评分和标注（有用/无用/部分正确）
- 支持用户在回答下方补充评论或纠正信息，反馈至知识库管理员
- 建立"知识纠错"工单系统，管理员可批量处理用户反馈并更新文档
- 支持多人协同提问与讨论，将问答过程转化为团队知识沉淀
