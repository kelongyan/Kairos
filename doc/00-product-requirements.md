# 可验证团队知识库与知识运营产品需求基准

---

## 1. 文档定位

本文档是 Kairos 新产品方向的需求基准，以根目录 `rag-multi-agent-kb.md` 为原始方案输入，并结合当前代码实现做了工程化取舍。后续 `doc/01-project-overview.md`、`doc/02-development-roadmap.md`、`doc/03-technology-stack.md` 和 `doc/04-development-progress.md` 均以本文档为上位依据。

原有“论文阅读与科研分析 Copilot”定位不再作为主线，而降级为智能知识库的一个垂直使用场景。原方案中的 Multi-Agent 也不作为第一阶段对外主卖点，而作为后续受控工作流能力。新的主线是：

```text
可验证的团队知识库问答与知识运营平台
  -> 多格式文档管理
  -> Hybrid RAG 精准检索
  -> 可溯源智能问答
  -> 用户反馈与知识缺口发现
  -> Trace / 评测 / 知识运营看板
```

---

## 2. 产品定位

Kairos 面向企业、团队和知识密集型组织，帮助用户把散落在 PDF、Word、Markdown、HTML、TXT、PPT 等文件中的内部知识转化为可检索、可问答、可追踪、可评估、可持续改进的知识资产。

核心目标不是做通用聊天机器人，也不是炫技式 Agent 平台，而是构建一个有证据、有权限、有审计、有反馈、有评测闭环的知识问答与知识运营平台。

一句话定位：

> Kairos 是一个可验证的团队知识库问答与知识运营平台。它用 RAG 把内部文档变成可追溯答案，用反馈、Trace 和评测持续发现知识缺口并改进知识质量。

核心原则：

- **知识库优先**：所有问答默认绑定到明确的知识库范围。
- **证据优先**：答案必须来自检索证据，证据不足时明确拒答或标记不确定。
- **知识运营优先**：不仅回答问题，还要沉淀热门问题、无答案问题、低质量引用和文档缺口。
- **可观测可评测**：检索、重排、生成、反馈、耗时和成本都需要可追踪。
- **受控 Agent 后置**：Agent 只在明确工作流和权限边界内执行规划、检索、分析、撰写和审核，不能作为短期核心卖点。
- **渐进产品化**：先复用当前 RAG 引擎，补齐知识库问答和运营闭环，再引入复杂 Agent。

## 3. 用户角色

| 角色 | 核心权限 |
|---|---|
| 系统管理员 | 管理用户、角色、全局配置、系统日志和平台级统计 |
| 知识库管理员 | 创建知识库，管理文档、切片策略、权限范围、问答质量和知识缺口 |
| 普通用户 | 在授权知识库内检索、问答、查看引用和提交反馈 |

第一版可以先不实现完整多租户和复杂组织架构，但数据模型应为后续 RBAC、团队和知识库隔离预留空间。

## 4. 核心功能范围

### 4.1 文档与知识库管理

目标能力：

- 创建、编辑、归档多个独立知识库。
- 文档归属于知识库，而不是全局散列表。
- 支持上传、查看、重建索引、删除文档。
- 支持 PDF 优先，后续扩展 DOCX、Markdown、HTML、TXT、PPTX。
- 文档状态保留 `uploaded -> parsing -> parsed -> indexing -> indexed | failed`。
- 文档删除或替换时同步清理或失效对应向量索引和 BM25 索引。

当前代码已具备 PDF 上传、解析、切片、embedding、Qdrant 索引和状态流转，但还没有知识库实体和文档归属关系。

### 4.2 Hybrid RAG 检索

目标能力：

```text
用户问题
  -> 查询改写
  -> dense retrieval + BM25 sparse retrieval
  -> RRF 融合
  -> rerank
  -> Evidence Pack
  -> LLM 基于证据回答
  -> citation + trace 返回
```

当前代码已经实现该链路的基础版本，但仍限制在单文档 `doc_id` 范围内。下一步应扩展为知识库级多文档检索。

### 4.3 智能问答

目标能力：

- 支持指定知识库范围问答。
- 支持多轮会话和短期上下文。
- 支持 SSE 流式输出。
- 每个答案返回引用、原文片段、页码或章节、检索分数。
- 证据不足时明确提示，不编造答案。

当前代码已有单轮、单文档问答和引用面板；多轮会话、SSE 和知识库范围尚未实现。

### 4.4 知识运营闭环

这是产品落地的核心差异点。系统不能只做“Chat with docs”，还要帮助知识库管理员持续改进知识资产。

目标能力：

- 记录用户问题、检索结果、答案、引用、反馈和证据不足原因。
- 支持用户反馈：有用/无用、引用准确/不准确、缺少关键内容、回答过长/过短。
- 识别知识缺口：无答案问题、低置信度问题、反复被问但引用不足的问题。
- 识别热门主题：高频问题、高频关键词、高访问文档。
- 识别低质量知识：引用不准确、重复文档、过期文档、解析失败文档。
- 为知识库管理员生成待处理清单：补充文档、更新文档、重新索引、创建 FAQ。

第一版不需要复杂大屏，但必须从数据模型开始保留这些运营信号。

### 4.5 Multi-Agent 协作

目标 Agent：

| Agent | 职责 |
|---|---|
| Planner | 判断任务复杂度，拆解步骤，选择工作流 |
| Retrieval | 调用本地知识库检索和后续外部工具 |
| Analyst | 对多份证据做对比、归纳和趋势分析 |
| Writer | 基于证据生成结构化回答或报告草稿 |
| Reviewer | 检查引用支撑、遗漏和潜在幻觉 |

Agent 不应在当前阶段立即实现。建议先完成知识库产品层、权限边界和 trace 持久化，再用 LangGraph 做受控编排。

### 4.6 可观测、评测与看板

目标能力：

- 每次问答生成 Trace ID。
- 持久化 query、rewrite、检索结果、Evidence Pack、回答、引用、耗时、模型和 token 成本。
- 支持固定评测集，衡量 Hit Rate、Recall@K、Citation Accuracy、Faithfulness、Latency。
- 支持知识库级统计：文档数、切片数、问答次数、高频问题、无答案问题、低质量引用、质量趋势。

当前代码已有结构化 retrieval trace 和最小评测 fixture，但还未持久化，也没有评测 API 和看板。

## 5. 非功能要求

- 安全：上传文档、网页内容和外部搜索结果全部视为不可信内容。
- 权限：所有知识库访问都必须为后续 RBAC 预留校验点。
- 可维护：业务层不得直接绑定具体 LLM、embedding、reranker 或向量库 SDK。
- 性能：第一版优先保证可重复运行和可调试；严格 P95 指标等进入生产化阶段再固化。
- 可降级：LLM 不可用时，应能返回检索片段或明确错误，而不是静默失败。

## 6. 技术采用边界

### 6.1 短期必须做

短期目标是把当前 RAG 引擎产品化为知识库系统，不引入过多新基础设施。

| 方向 | 采用内容 | 说明 |
|---|---|---|
| 知识库产品层 | `KnowledgeBase`、文档归属、多文档检索 | 下一阶段主任务 |
| 知识运营闭环 | 问题记录、反馈、无答案问题、低质量引用 | 下一阶段必须预留 |
| 检索 | 继续 `Qdrant dense + rank-bm25 + RRF + reranker boundary` | 保持现有链路 |
| Trace | 先自建 PostgreSQL trace 表 | 不立即接外部平台 |
| 权限 | 最小 JWT + RBAC 数据模型 | 先满足知识库访问控制 |
| Reranker | 增加真实 reranker provider | 保留 simple fallback |
| 前端 | 知识库切换、文档分组、trace 展示 | 不重做整体 UI |

### 6.2 中期可引入

这些能力在知识库产品层稳定后引入。

| 工具 / 技术 | 用途 | 引入条件 |
|---|---|---|
| LangGraph | 受控 Multi-Agent 工作流 | trace、权限、知识库级检索稳定后 |
| Ragas / DeepEval | RAG 评测 | 固定评测集和 trace 表建立后 |
| Langfuse / Phoenix | LLM 观测、trace、成本分析 | 自建 trace 字段稳定后 |
| MarkItDown | 快速扩展 DOCX/PPTX/HTML 到 Markdown | PDF 主链路稳定后 |
| Unstructured | 多格式文档结构化切分 | 需要更细粒度元素类型时 |
| BGE / Jina / Cohere / Voyage reranker | 更可靠重排 | simple fallback 无法满足质量要求时 |

### 6.3 长期升级评估

这些方向只在规模、复杂度或产品需求明确出现后评估。

| 工具 / 技术 | 暂缓原因 | 升级触发条件 |
|---|---|---|
| Docling | 能力强但解析链路更重 | 表格、公式、OCR、复杂 PDF 成为核心需求 |
| Elasticsearch / OpenSearch | 运维成本高 | BM25、过滤、全文检索成为主要瓶颈 |
| Milvus | 更适合大规模向量集群 | Qdrant 在容量或并发上不足 |
| Qdrant sparse vector / 原生 hybrid | 需要调整索引设计 | Python BM25 成为性能瓶颈 |
| LiteLLM Proxy | 会引入模型网关运维 | 模型供应商、限流、预算、fallback 复杂化 |
| Casbin / OpenFGA | 权限模型复杂度高 | 简单 RBAC 无法表达团队、继承、共享关系 |
| MCP 工具层 | 工具治理复杂 | 出现多个稳定外部工具接入需求 |
| A2A | 跨系统 Agent 协议仍属后期能力 | 需要和其他 Agent 系统协作 |
| GraphRAG | 建模和评测成本高 | 跨文档实体关系和多跳推理成为核心场景 |

---

## 7. 明确暂缓

以下能力不是下一阶段优先项：

- 完整 GraphRAG。
- A2A 跨系统 Agent 协作。
- MCP 工具市场。
- 开放式自主 Agent。
- 模型微调。
- 移动端和企业 IM 接入。
- 大规模分布式部署。
- 全量多模态文档理解。
- 复杂大屏看板。
- 自动写入、删除、修改外部系统数据。

这些方向可以保留在长期扩展规划中，但不应阻塞知识库产品层落地。
